# 바른손카드 ↔ 바른라운지 파트너 어드민 연동 명세 (초안)

작성일: 2026-07-07 · 개발자 미팅용
목표: 바른손카드 사이트에서 로컬딜 이용권을 판매(예시 UX: https://experiences.myrealtrip.com/products/6031614 형태의 딜 상세)하고, **쿠폰 발급 → 파트너 어드민 사용처리 → 사용 기준 정산(익월 25일)**이 실시간 연동되는 것.

프로토타입: `C:\src\barunson-deal` (Vite+TS, `npm run dev`). 연동 지점은 `src/api.ts`·`src/auth.ts`에 격리되어 있으며, 함수 시그니처를 유지한 채 내부만 `fetch()`로 교체하는 구조.

## 1. 시스템 역할 분담 (제안 — 미팅에서 확정)

| 영역 | 바른손카드(본사 시스템) | 파트너 어드민 |
|---|---|---|
| 딜 게재·판매·결제 | ✅ 소유 | — |
| 쿠폰 발급·코드 채번 | ✅ 소유 (단일 진실원천) | 조회만 |
| 쿠폰 상태(사용/환불) | ✅ 소유 — 중복 사용 방지의 기준 | 사용처리 요청 |
| 정산 계산·지급 | ✅ 소유 (수수료율 적용, 지급 배치) | 조회(예정액·내역·명세서) |
| 고객 알림(문자/알림톡) | ✅ 발송 게이트웨이 | 발송 트리거 |
| 업체·지점·계정 관리 | 마스터 보유 | 신청·승인 UI |

핵심 원칙: **쿠폰 상태와 정산 금액의 계산 주체는 본사 시스템 1곳**. 어드민은 뷰 + 사용처리 트리거.

## 2. 도메인 모델 (프로토타입 `src/types.ts` 기준)

- **Coupon**: `code, type('service'|'amount'|'count'), product, name, phone, buyDate, expire, paid(결제가=정산기준), origin?, face?(액면가|총회수), used?, status, branchId(구매지점), usedBranchId?, history[]`
- **UsageEntry**: `date, amount(원|회), branchId(처리지점=정산귀속)`
- **Branch**: `id, name, kind('direct'|'franchise'), payee(정산주체)`
- **정산 규칙** (PRD §5.1~5.2):
  - 사용 기준: 매월 1일~말일 사용처리분 → 익월 25일 지급(휴일 시 직전 영업일). 미사용분 정산 없음.
  - 일반형: 판매가 전액 / 금액형: 차감액면 × (결제액÷총액면) / 횟수형: 회수 × 회당단가(결제액÷총회수), 완주 시 끝수 보정.
  - **다지점**: 전 지점 사용 가능(동일 상품·동일가 전제), 정산은 **사용처리한 지점** 귀속. 직영=본사 합산, 가맹=각 가맹점 계좌 직접 지급, 세금계산서 사업자별.

## 3. 필요한 API (어드민 → 본사)

프로토타입 목 함수와 1:1 매핑. Base: `https://api.barunn.example/partner/v1` (가칭)

| # | 프로토타입 함수 | 제안 엔드포인트 | 비고 |
|---|---|---|---|
| 1 | `api.listCoupons()` | `GET /coupons?vendorId&branchId&status&query&page` | 검색(코드/연락처 끝4자리/이름) 서버측 지원 필요 |
| 2 | `api.redeem(code, branchId)` | `POST /coupons/{code}/redeem` body `{branchId}` | **멱등키 필수**, 409=이미 사용 |
| 3 | `api.deduct(code, amount, branchId)` | `POST /coupons/{code}/deduct` body `{amount, branchId}` | 금액형=원, 횟수형=회. 잔여 초과 시 422 |
| 4 | `api.sendSms(...)` | 본사에서 사용처리 이벤트에 후킹해 자동 발송 권장 | 어드민 직접 호출 불필요해질 수 있음 |
| 5 | `api.listSms()` | `GET /notifications?vendorId` | 발송 이력 |
| 6 | `settleLines`/정산 화면 | `GET /settlements?vendorId&month` + `GET /settlements/next` | 서버가 계산(수수료 공제 포함)해 내려주는 게 정석 |
| 7 | 명세서 CSV | `GET /settlements/{payoutDate}/{payeeId}/statement` | 지급일×정산주체 단위 |
| 8 | `api.listNotices()` 등 | `GET /notices` | 공지 |
| 9 | `auth.*` (로그인·권한신청·승인) | 본사 계정 체계에 편입: `POST /auth/login`, `POST /vendor-access`, 직원승인 API | 아래 §5 |
| 10 | (신규) 환불 반영 | 웹훅 `coupon.refunded` 또는 폴링 | 어드민에 환불 상태 표시용 |

**동시성/멱등성 요구사항 (중요)**
- 사용처리는 두 지점에서 동시에 눌러도 1건만 성공해야 함 → 서버에서 상태 전이를 원자적으로 처리 + `Idempotency-Key` 헤더.
- 사용처리 응답에는 처리 후 상태(잔액/잔여회수, 처리일시, 처리지점)를 포함해 어드민이 재조회 없이 영수증 표시.

## 4. 판매(발급) 시 본사 → 쿠폰 데이터 요건

딜 상세(마이리얼트립형)에서 구매 시 생성되어야 하는 필드:
- 상품·유형(일반/금액형/횟수형)·판매가·정상가·액면가/총회수
- **지점 선택**: 구매 시 판매 지점 지정 — 사용 범위는 해당 지점 운영 사업자의 지점 내(직영 계열 교차 사용, 가맹 전용)
- 유효기간(딜별), 고객명·연락처(수신 동의), 쿠폰 코드 채번 규칙
- 옵션/회차 구성이 있는 딜(예시 링크처럼 옵션 선택형)의 경우 옵션→상품 매핑 방식

**소비자 사이트 부속 요건 (환불·만료 컴플라이언스 — 기능_PRD §2.5):**
- 만료 D-30 자동 알림(잔액·만료일·연장/환급 안내), 마이페이지 환불·연장 신청 버튼
- 만료 후에도 구매일+5년까지 90% 환급 신청 가능(만료 쿠폰 마이페이지 유지)
- 환불 확정 시 파트너 어드민으로 상태 전파(웹훅 `voucher.refunded`)

## 5. 계정·권한 모델 (§실운영)

- 역할 3층: 바른손카드 직원 / 브랜드 관리자(본사) / **지점 계정(branchId 고정)**.
- 지점 계정 로그인 시 처리 지점 자동 고정(현 데모의 셀렉터는 제거), 자기 지점 쿠폰·정산만 조회.
- 온보딩: 브랜드 관리자가 지점 등록 → 지점 담당자 초대 → 가맹점주는 정산 계좌·사업자 정보 직접 등록 → 바른손카드 승인.
- 개인정보 원칙(현 프로토타입 반영됨): 미사용 쿠폰의 고객 PII는 업체에 비노출, 사용처리 시점에만 표시. 연동 시에도 서버 응답 레벨에서 필터링 권장.

## 6. 미팅에서 결정할 것 (Open Questions)

1. **쿠폰 코드 채번·상태 관리 주체** — 본사 커머스 DB인지 별도 쿠폰 서비스인지.
2. **정산 계산 주체** — 서버 계산(권장) vs 어드민 계산. 수수료율(계약 제5조, 요율 미정) 어디서 관리?
3. **가맹점 직접 지급** — 지급 배치·세금계산서 발행 시스템이 정산주체(사업자) 복수 지급을 지원하는지.
4. **인증 방식** — 본사 통합 계정(SSO/JWT)인지 파트너 전용 계정인지. 직원 Google 로그인 유지 여부.
5. **알림 발송** — 문자 vs 카카오 알림톡, 발송 주체(본사 게이트웨이), 템플릿 사전 승인 일정.
6. **환불 이벤트 전달** — 웹훅 vs 폴링. 어드민에서 환불 표시 요건.
7. **딜 옵션 구조** — 옵션 선택형 딜을 쿠폰 유형 3종에 어떻게 매핑할지.
8. **배포 형태** — 파트너 어드민을 정적 호스팅(dist/) + API 도메인 CORS인지, 본사 도메인 서브패스인지.
9. **일정/환경** — 스테이징 환경, 테스트 계정, API 문서(OpenAPI) 제공 여부.
