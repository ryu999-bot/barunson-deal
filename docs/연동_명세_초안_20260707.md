## 2. 도메인 모델 (프로토타입 `src/types.ts` 기준)

- **Coupon**: `code, type('service'|'amount'|'count'), product, name, phone, buyDate, expire, paid(결제가=정산기준), origin?, face?(액면가|총회수), used?, status, branchId(구매지점), usedBranchId?, history[]`
- **UsageEntry**: `date, amount(원|회), branchId(처리지점=정산귀속)`
- **Branch**: `id, name, kind('direct'|'franchise'), payee(정산주체)`
- **정산 규칙** (PRD §5.1~5.2):
  - 사용 기준: 매월 1일~말일 사용처리분 → 익월 25일 지급(휴일 시 직전 영업일). 미사용분 정산 없음.
  - 일반형: 판매가 전액 / 금액형: 차감액면 × (결제액÷총액면) / 횟수형: 회수 × 회당단가(결제액÷총회수), 완주 시 끝수 보정.
  - **다지점**: 사용 범위 = **판매 지점을 운영하는 사업자(정산주체)의 지점 내**(직영 계열 교차 사용, 가맹점 쿠폰은 해당 가맹점 전용). 정산은 **사용처리한 지점** 귀속. 직영=본사 합산, 가맹=각 가맹점 계좌 직접 지급, 세금계산서 사업자별.

## 3. 확정 API 매핑 (핸드오버 기준 — 어드민 목 함수 → bar_shop1 PublicApi)

인증: `POST /api/Partner/authenticate` (clientId `thelounge` + secret) → Bearer 토큰(+refresh). 토큰 스코프 = 파일럿 업체(정산주체), 서버가 스코프 강제(403).

| # | 프로토타입 함수 | 확정 엔드포인트 | 비고 |
|---|---|---|---|
| 1 | `api.listCoupons()`/검색 | `GET /api/Lounge/vouchers/search?code|phone4|name` | 서버측 검색·PII 마스킹·스코프 필터 ✅ 제공 |
| 2 | `api.redeem(code, branchId)` | `POST /api/Lounge/vouchers/{code}/redeem` body `{branchId}` | **`Idempotency-Key` 헤더 필수**, 409=이미 사용 ✅ |
| 3 | `api.deduct(code, amount, branchId)` | `POST /api/Lounge/vouchers/{code}/deduct` body `{branchId, amount}` | 금액형=원/횟수형=회, 422=잔여 초과 ✅ |
| 4 | (신규) 지점 관리 | `POST/GET/PUT /api/Lounge/branches` | 지점 등록·목록·수정·사용중지 — payee는 토큰 스코프 자동 ✅ |
| 5 | `api.sendSms(...)` | **API 없음** — 사용/차감 성공 시 서버가 자동 발송 | 어드민 호출 제거 |
| 6 | `auth.*` (로그인·계정 발급/회수) | **어드민 자체** (bar API 아님) | 발급제 직원 콘솔 유지 |

**제공 완료(2026-07-13):** 사용 원장 조회(`GET /api/Lounge/usages`), 발급 목록 조회(`GET /api/Lounge/vouchers`).
**미제공 → 회신 필요 (§6):** 정산 ERP 직접조회, 문자 이력, 환불 웹훅.

**동시성/멱등성 (핸드오버로 확정)**
- 두 지점 동시 처리에도 1건만 성공(서버 원자 처리), 재시도는 같은 `Idempotency-Key`로 안전.
- 응답에 처리 후 상태 포함: redeem → `{usageId, status}`, deduct → `{usageId, balance, status}`.

## 4. 판매(발급) 시 본사 → 쿠폰 데이터 요건

딜 상세(마이리얼트립형)에서 구매 시 생성되어야 하는 필드:
- 상품·유형(일반/금액형/횟수형)·판매가·정상가·액면가/총회수
- **지점 선택**: 구매 시 판매 지점 지정 — 사용 범위는 해당 지점 운영 사업자의 지점 내(직영 계열 교차 사용, 가맹 전용)
- 유효기간(딜별), 고객명·연락처(수신 동의), 쿠폰 코드 채번 규칙
- 옵션/회차 구성이 있는 딜(예시 링크처럼 옵션 선택형)의 경우 옵션→상품 매핑 방식

**소비자 사이트 부속 요건 (환불·만료 컴플라이언스 — 기능_PRD §2.5):**
- 만료 D-30 자동 알림(잔액·만료일·연장/환급 안내), 마이페이지 환불·연장 신청 버튼
- 만료 후에도 구매일+5년까지 90% 환급 신청 가능(만료 쿠폰 마이페이지 유지)
- 환불 확정 시 파트너 어드민으로 상태 전파(웹훅 `voucher.refunded`)

## 5. 계정·권한 모델 (사업자 회원가입 + 직원 승인제 — 2026-07-09 개정)

- **역할 2종**: 바른손카드 직원(staff) / 업체 사업자 계정(vendor).
- **회원가입 = 사업자(정산주체) 단위**: 사업자명·사업자번호·사업자등록증 제출 → **승인 대기** → 직원이 통합관리자 제휴사 정보와 대조 후 승인(반려 시 계정 삭제). 승인 전에는 메뉴 잠금. 직원 직접 발급 경로도 병행(즉시 사용).
- **지점 스코프**: 계정 소속 지점 = 발급 시 지정분(branchIds) + 지점 관리(F-13)에서 자기 사업자(payee)로 등록한 지점. 지점 1개면 처리 지점 고정, 0개면 사용처리 전 등록 안내. 조회·사용처리·정산 범위가 계정으로 결정.
- **로그인 계정 관리 = 어드민 자체**(bar API 범위 아님 — 핸드오버 §4). 제휴사·수수료율 마스터는 통합관리자.
- 비밀번호 재설정은 직원 콘솔/고객센터 경유(연동 시 서버 정책, 해시 저장).
- 개인정보 원칙(현 프로토타입 반영됨): 미사용 쿠폰의 고객 PII는 업체에 비노출, 사용처리 시점에만 표시. 연동 시에도 서버 응답 레벨에서 필터링 권장.

   - (2026-07-13 계약서 법무본) 정산 상태 **지급 보류(HOLD)** — 계약서 제8조의2 ⑧(을↔가맹점 분쟁 시 갑이 정산금 지급 보류). 상태 `PENDING/CONFIRMED/**HOLD**/PAID`. **지급 배치·보류 판정은 바른 ERP에서 직접 반영**, 어드민은 보류 배지 표시만.
3. ✅ **해소 — 발급 목록 API 제공**: `GET /api/Lounge/vouchers`(상태·지점 필터, 페이지). 프론트 `listCoupons` 연동됨. 판매현황 KPI에 사용.
4. ❌ **미해소 — 문자 발송 이력**: 문자 자동발송이 아직 서버 미구현(TODO) → 이력 조회 API도 없음. 발송 구현 시 `TB_Lounge_Sms_Log` 기반 조회 API 제공. **F-06 화면은 당분간 보류/축소.**
5. ❌ **미해소 — 환불 이벤트 전달**: 환불 흐름 미구현. 현재 REFUNDED는 `vouchers` 조회 시점에만. 웹훅(`voucher.refunded`)은 환불 기능 구현 시 제공.
6. ✅ **해소 — 멀티 테넌트**: 클라이언트는 `thelounge` 1개 유지. `POST /api/Lounge/auth/scope`에 업체 `CompanySeq`(제휴사관리 등록값)를 넣으면 그 업체 스코프 토큰 발급. 업체 추가 시 credential 추가 발급 불필요.
